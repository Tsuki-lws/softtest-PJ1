# 软件测试计划

## 1 目的

本计划书用于安排场馆预约管理系统（`SoftwareTestingDemo`）课程作业的测试工作。测试工作主要分为两部分：一是对 `controller` 接口做集成测试，二是对 `service.impl` 实现类做单元测试，检查系统主要业务功能是否正常、业务流程是否顺畅，并识别测试过程中可能出现的环境、进度和协作风险。

- 测试范围：用户、场馆、订单、新闻、留言以及管理员相关功能
- 测试方法：黑盒测试与白盒测试结合
- 测试资源：测试人员、JDK、Maven、H2、JUnit 5、Mockito、MockMvc、JaCoCo
- 测试时间：2026-04-10 至 2026-04-26
- 主要风险：人员协作、环境依赖、进度协调、文档与测试结果口径不一致

## 2 测试计划标识

`PJ1_20260410_场馆预约管理系统`

## 3 介绍

场馆预约管理系统是一个面向普通用户和管理员的 Web 管理系统。普通用户主要使用前台页面完成浏览、预约和留言，管理员主要使用后台页面完成数据维护和审核。

- 普通用户可以注册登录、浏览场馆信息、查看新闻、预约场馆、查看和管理订单、发表留言
- 管理员可以管理用户、场馆、新闻、订单和留言审核

系统主要功能模块如下：

1. 用户注册登录模块  
   用户可以注册和登录系统，管理个人信息和预约信息；管理员可以对用户信息进行增删改查。

2. 场馆信息展示模块  
   用户可以查看场馆位置、开放时间、价格、图片等信息；管理员可以对场馆进行增删改查。

3. 预约订单管理模块  
   用户可以选择场馆和时间进行预约，并查看、修改、删除自己的订单；管理员可以对订单进行审批和管理。

4. 新闻展示模块  
   用户可以查看新闻资讯和新闻详情；管理员可以新增、修改和删除新闻。

5. 留言板模块  
   用户可以留言；管理员可以审核、拒绝和删除留言。

## 5 要测试的特性

### 5.1 管理员端集成测试要测试的特性

3. 订单管理特性
   - 订单审核功能：重点检查通过、拒绝后的状态变化是否正确
   - 订单列表展示功能：检查分页查询和后台订单列表是否正常

4. 用户管理特性
   - 用户增删改功能：检查新增、修改、删除后的跳转结果和数据库更新情况
   - 用户列表展示功能：检查分页查询和用户列表展示是否正常
   - 用户 ID 检查功能：检查已存在和未存在用户 ID 的返回结果
   - 后台页面访问控制：检查未登录情况下访问后台用户管理页面的处理方式

5. 场馆管理特性
   - 场馆增删改功能：检查新增、修改、删除后的跳转结果和数据变更情况
   - 场馆列表展示功能：检查分页查询和场馆列表展示是否正常
   - 图片上传功能：检查上传后图片路径是否成功保存
   - 场馆名称检查功能：检查已存在和未存在场馆名称的返回结果

### 5.2 用户端集成测试要测试的特性

1. 用户管理特性
   - 页面导航功能：检查注册页、登录页、用户信息页等页面能否正常访问
   - 用户认证功能：检查普通用户登录、管理员登录、重复注册和未登录访问限制
   - 用户状态管理功能：检查登出后的会话清理和页面跳转
   - 用户信息更新功能：检查个人资料修改后的返回结果和数据变化
   - 密码相关检查：检查密码校验接口在正常用户、错误密码和不存在用户三种情况下的返回表现
   - 注册数据处理：检查注册后密码字段的保存方式是否符合预期

2. 留言管理特性
   - 留言页面访问功能：检查已登录和未登录两种情况下的访问控制
   - 留言获取功能：检查公开留言列表和个人留言列表的查询结果
   - 留言操作功能：检查发送、修改、删除留言后的状态和数据变化

3. 订单管理特性
   - 订单页面访问功能：检查订单页、下单页、修改页和未登录访问控制
   - 订单数据获取功能：检查订单分页查询和按场馆查询的返回结果

   - 输入边界与鉴权：检查负数预约时长、未登录删除订单等场景下系统的处理结果

4. 新闻管理特性

   - 新闻列表获取功能：检查分页查询和默认分页参数处理
   - 分页边界处理：检查页码为 0 等边界输入下系统的处理结果

## 6 不会被测试的特性

以下内容不在本计划范围内：

1. 性能测试、压力测试和稳定性测试：这类测试需要专门的压测工具和稳定环境，本次课程作业不展开。
2. 安全性专项测试：不进行渗透测试、漏洞扫描、XSS/SQL 注入等专项安全测试，只在功能测试中覆盖部分鉴权和输入边界场景。
3. 移动端兼容性测试和多浏览器全量兼容性测试：本次测试以 Web 端主要浏览器环境为主，不覆盖全部设备和浏览器组合。
4. 并发场景和高并发预约冲突测试：测试环境资源有限，不模拟大量用户同时预约。
5. 故障恢复、业务连续性和数据库备份恢复测试：相关场景需要额外环境和恢复策略，不列入本轮测试。
6. 真实 MySQL 环境下的部署验证：本次自动化测试使用 H2 内存数据库，不进行真实 MySQL 部署验证。

本计划只覆盖系统核心功能和主要业务流程，不展开性能、安全和并发等专项测试。

## 7 方法

### 7.1 测试策略与方法

本次工作同时采用黑盒测试和白盒测试。

- 黑盒测试：使用等价类划分、边界值分析、错误推测法和简单决策表方法，重点验证页面跳转、接口返回、会话状态和数据库变化是否符合预期。
- 白盒测试：使用语句覆盖、判定覆盖和关键分支覆盖，重点验证 `service.impl` 业务逻辑、状态流转和异常分支。

### 7.2 主要活动

1. 分析项目需求并确定测试范围
2. 搭建测试环境并准备测试数据
3. 编写测试用例和测试代码
4. 执行测试并记录结果
5. 分析缺陷并整理测试结论
6. 输出测试计划书、测试用例设计文档和测试总结

### 7.3 测试技术与工具

- JUnit 5：测试框架与断言
- Mockito：模拟依赖，隔离单元测试
- Spring Boot Test：启动测试上下文
- MockMvc：模拟 HTTP 请求，进行接口级集成测试
- H2：提供内存数据库测试环境
- JaCoCo：生成覆盖率报告
- Maven Surefire：生成测试执行报告
- JsonPath：检查 JSON 响应内容
- Hamcrest / AssertJ：补充断言表达能力

### 7.4 测试广度和深度

- 测试广度：覆盖主要业务模块的正常流程与常见异常流程
- 测试深度：对登录、预约、审批、留言审核、分页查询和文件上传等关键业务进行边界和分支验证

### 7.5 完成准则

- 在 `SoftwareTestingDemo/SoftwareTestingDemo` 目录下执行 `mvn test`，能够完整输出测试执行结果
- 计划范围内的核心功能点均有对应测试用例和测试脚本
- 能正常生成 Surefire 执行报告和 JaCoCo 覆盖率报告
- 对执行过程中发现的失败用例、异常用例和缺陷现象进行记录与分析
- 若存在未解决缺陷，已说明影响范围和后续处理建议
- 计划书、测试用例文档和测试总结在范围、编号和结论口径上保持一致

### 7.6 需求与测试项对应方式

本项目没有单独引入需求管理平台，需求与测试项的对应关系主要通过以下方式维护：

- 以系统现有功能模块为主线划分测试项，包括用户、场馆、订单、新闻、留言和后台管理功能
- 以 `src/test/java` 中的测试类和测试方法作为测试落实依据
- 通过共享文档维护测试用例编号、功能点和预期结果
- 以 Surefire 报告和 JaCoCo 报告作为执行结果与覆盖情况的核对依据
- 以《测试用例设计文档》和《测试总结》作为测试过程和结果的书面记录

### 7.7 测试约束

- 部分组员本机可能没有完整的 Maven 环境
- 本轮工作不包含性能、安全和并发专项

## 8 测试项通过/失败准则

### 8.1 通过准则

- 对应测试脚本能够正常执行，断言结果与预期一致
- 页面跳转、接口返回、数据库状态变化与测试预期一致
- 执行过程中没有出现导致用例中断的未处理异常
- 对边界输入和异常输入的处理符合用例预期

### 8.2 失败准则

- 测试脚本执行失败，或者断言结果与预期不一致
- 页面跳转、接口返回或数据库结果与预期不符
- 核心业务流程无法继续执行，或者出现未处理异常
- 系统没有正确处理计划范围内的边界输入或异常输入

## 9 暂停准则和继续准则

### 9.1 暂停准则

当出现以下情况时暂停测试：

| 暂停情况 | 原因说明 |
| --- | --- |
| 核心功能测试无法继续 | 登录、预约、审核等关键功能异常会影响后续用例执行 |
| 项目无法编译或启动 | 编译失败、依赖缺失或 Spring 上下文无法加载时，测试结果不可信 |
| 测试环境不可用 | Maven、JDK、H2 配置或测试资源异常会影响测试执行 |
| 测试数据污染 | 用户、场馆、订单等基础数据异常会导致结果不可复现 |
| 测试代码与被测代码版本不一致 | 文档、测试脚本和源码版本不一致会导致结论失真 |
| 计划执行偏离 | 如果实际进度明显偏离时间表，需要重新调整任务优先级 |

### 9.2 继续准则

满足以下条件后继续测试：

| 暂停情况 | 继续准则 |
| --- | --- |
| 核心功能测试无法继续 | 相关缺陷修复后，关键用例复测通过 |
| 项目无法编译或启动 | 编译和启动问题解决后，重新执行基础测试 |
| 测试环境不可用 | 环境配置恢复后，重新执行受影响用例 |
| 测试数据污染 | 测试数据重新初始化或修复后，再继续执行测试 |
| 测试代码与被测代码版本不一致 | 统一代码基线、测试脚本和文档版本后继续测试 |
| 计划执行偏离 | 调整计划并确认剩余任务优先级后继续推进 |

## 15 风险以及应急措施

1. 人员协作风险  
   组员分工虽然已经明确，但如果沟通不及时，容易出现文档、代码和测试结果不同步的问题。  
   处理方式：通过共享文档和代码仓库统一进度，关键修改在组内同步，提交前集中做一次交叉检查。

2. 环境风险  
   部分机器可能缺少 Maven，或者依赖下载不稳定。  
   处理方式：由环境完整的机器统一执行测试，并保留测试报告。

3. 进度协调风险  
   测试代码、用例文档和总结文档并行推进时，如果节奏不一致，后期容易出现口径差异。  
   处理方式：提前留出互审时间，提交前集中核对一次。

4. 口径风险  
   文档内容和测试结果可能出现不一致。  
   处理方式：以测试脚本、Surefire 报告和 JaCoCo 报告为准进行复核。

5. 范围风险  
   如果临时扩大测试范围，文档和测试代码都可能失去控制。  
   处理方式：只保留核心功能测试，不额外扩展到性能、安全、并发专项。

6. 技术风险  
   组员对 Spring Boot Test、MockMvc、Mockito 等测试工具的熟悉程度不同，可能导致测试脚本风格不统一，排查问题时效率偏低。  
   处理方式：以现有测试脚本为统一参考，编写和修改测试时尽量沿用已有写法，遇到分歧时先统一规则再继续推进。
