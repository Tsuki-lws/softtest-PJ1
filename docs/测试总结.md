### 1.3 测试方法策略

**计划测试策略方法：**

1. 黑盒与白盒结合。

2. 集成测试：`Spring Boot Test + MockMvc + H2`，验证控制层端到端链路。

**结论：**

实际执行策略与计划一致。

### 1.4 测试风险

**计划风险：**

1. 环境不稳定导致结果波动。
2. 数据依赖导致用例不可重复。
3. 分支遗漏导致结论偏差。

**实际发生风险：**

1. 缺少浏览器 UI 自动化，前端交互场景覆盖不足。
2. 默认 `mvn test` 中已经复现出 6 个问题场景，说明系统仍存在产品级残留风险。

**风险分析：**

1. 当前自动化以后端链路为主，无法完全替代端到端 UI 验证。
2. 文件系统与异常类场景已补充专项测试，后续重点转向 UI 与业务规则完善。

**应对建议：**

1. 增加关键流程 UI 自动化回归。
2. 在 CI 中增加覆盖率阈值门槛。
3. 对非法业务参数增加产品级校验与回归用例。

## 4 经验总结

### 好的经验

1. 采用“单元测试 + 集成测试 + JaCoCo”形成质量闭环。
2. 自动化回归可快速定位问题并提升重复验证效率。
3. 边界、鉴权与安全场景用例帮助我们发现了覆盖率之外的真实业务风险。

---

### 需要改进的点

1. 当前覆盖率尚未设置强制门槛。
2. 关键业务流程缺少 UI 自动化验证。
3. 后台权限、安全性和边界参数的产品级约束仍需继续补强。

**问题分析：**

仅依赖后端自动化测试，难以覆盖真实交互场景与端到端风险。

**改进建议：**

1. 在 CI 中增加覆盖率阈值校验。
2. 增加关键流程 UI 自动化回归。
3. 对非法业务参数补充产品级校验与回归测试。
